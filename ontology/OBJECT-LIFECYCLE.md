# LOGOS Object Lifecycle

This document defines how LOGOS objects mature over time.

The system must distinguish between a raw idea, a tested hypothesis and a validated law.

---

## 1. Why Lifecycle Matters

Without lifecycle discipline, LOGOS becomes a collection of attractive thoughts.

With lifecycle discipline, LOGOS becomes a learning system.

A sentence is not automatically a law.

A good script is not automatically a belief shift.

A viral video is not automatically proof.

Each object must move through maturity stages.

---

## 2. Core Statuses

```text
raw
candidate
draft
testing
validated
rejected
archived
law_candidate
law
```

---

## 3. raw

A raw object is an unprocessed thought.

Examples:

- a note from a conversation;
- a possible insight;
- a phrase;
- a tension noticed in an audience;
- a possible story direction.

Raw objects should not be treated as strategy.

---

## 4. candidate

A candidate object is promising enough to formalize.

It has a possible role in the system but is not yet shaped.

Example:

```text
Maybe the core contradiction is visible care vs invisible care.
```

---

## 5. draft

A draft object has structure but has not yet been tested.

A draft should follow the correct schema.

Examples:

- drafted Human Truth;
- drafted Belief Shift;
- drafted Meaning Atom;
- drafted script;
- drafted experiment.

---

## 6. testing

A testing object is being used in real experiments or controlled internal evaluation.

Examples:

- a script is published;
- a Meaning Atom is tested across multiple formats;
- a Story Pattern is compared against another pattern.

Testing requires metrics or structured feedback.

---

## 7. validated

A validated object has enough evidence to be used confidently inside its current context.

Validated does not mean universal.

A DeflaGyn-validated object may still be only runtime-specific.

---

## 8. rejected

A rejected object failed because it was unclear, unsafe, weak, misleading or unsupported by evidence.

Rejected objects should not be deleted automatically.

They may contain useful learning.

---

## 9. archived

An archived object is no longer active but is kept for historical memory.

Archive when:

- the context changed;
- the idea was replaced;
- the experiment is no longer relevant;
- the object is a duplicate.

---

## 10. law_candidate

A Law Candidate is a repeated pattern that may become a LOGOS Law.

It must have evidence from more than one experiment or context.

A Law Candidate is still not a law.

---

## 11. law

A LOGOS Law is a validated principle that improves future decisions.

A law must be:

- clear;
- reusable;
- evidence-supported;
- useful for generation;
- not limited to one lucky case.

---

## 12. Default Maturity Path

```text
raw
→ candidate
→ draft
→ testing
→ validated
→ law_candidate
→ law
```

Alternative paths:

```text
raw → rejected
candidate → archived
testing → rejected
validated → archived
```

---

## 13. Runtime-Specific vs Universal

A runtime-specific object can be validated inside one client.

A universal object must prove relevance beyond one client or category.

Example:

```text
Runtime-specific:
A Ukrainian women's health phrase works for DeflaGyn.

Universal:
A contradiction-based reframe improves attention across multiple categories.
```

---

## 14. Promotion Rules

An object can be promoted only when it has:

1. clear structure;
2. traceable relationships;
3. documented evidence;
4. defined context;
5. a reason for promotion.

---

## 15. Demotion Rules

An object can be demoted when:

- new evidence contradicts it;
- it causes unsafe communication;
- it creates confusion;
- it is too client-specific for universal use;
- it duplicates a stronger object.

---

## 16. Lifecycle Ownership

Every active object should have a responsible issue or file.

If an object matters, it must be visible in the system.

If it is invisible, it cannot be managed.

---

## 17. Final Rule

A LOGOS object is not mature because it sounds good.

It is mature because it is structured, traceable, tested and useful.
